A safety net for AI-edited n8n · no build step · MIT

Let AI edit your n8n —
without losing work.

AI agents build n8n flows fast — and can just as fast overwrite a node or break a workflow you can't rebuild. n8c keeps your workflows, prompts and credentials as versioned code, deploys them with a plan / apply loop, and rolls the whole instance back to a known-good release in one command.

Get started ★ View on GitHub 
# snapshot → let AI edit → review → ship → roll back if needed
$ n8c pull              # live n8n → versioned files
$ n8c plan              # diff files vs live — no surprises
$ n8c apply             # deploy; every apply is a release
$ n8c restore <rel>   # broke something? one-command rollback
Why n8c

The UI is great for building. Undo is the hard part.

n8n keeps per-workflow revisions, but with a pile of history entries, no diff, no cross-workflow view, and no way to roll the whole instance back to a known-good point, undoing a bad change — especially one an AI made — is painful. n8c is the safety net.

🤖

A safety net for AI

Let an AI edit your automation freely. Every apply is a restorable release, so n8c restore <release> returns the whole instance — workflows, prompts and credentials together — to a point you know was good.

🙈

Secrets your AI never sees

Credential values live only in your .env, resolved from process.env at deploy time — never in a file, never committed, never in the directory the agent works in. Code holds only process.env.MY_TOKEN.

🔐

Encrypted at rest

Before a credential reaches the database it's encrypted with AES-256-GCM (N8C_CREDENTIAL_ENCRYPTION_KEY) — never persisted in plaintext.

🧭

Staging → prod, same files

Entities use env-neutral IDs and n8c auto-maps each environment's real n8n and credential IDs. Build on staging, then n8c -e prod apply the exact same files — no hand-editing IDs.

📐

Real plan / apply

plan diffs your files against live n8n and freezes the actions; apply pushes to n8n before recording success, so a failed deploy never leaves a false "deployed" state.

⚡

Lightweight, no server needed

Install in seconds — dependency-light (just commander + your DB driver). Store history in MongoDB for teams, or in a single SQLite file (built-in node:sqlite) with zero setup.

Before you start

Requirements

Node.js ≥ 22.18Or ≥ 23.6. n8c reads your project's .ts entity files at runtime using Node's built-in type stripping, on by default from these versions. On 22.6–22.17 use the --experimental-strip-types flag.
A databaseMongoDB (teams / shared instances) or SQLite (a single local file, zero setup via built-in node:sqlite). See Storage backends.
n8nWith the Public API enabled (API 1.1.x) and an API key.
Quickstart

From zero to deployed in five steps

Install once, point n8c at your n8n instance and a database (MongoDB or SQLite), and you're running the full loop.

npm install -g n8c
# …or run without installing:  npx n8c --help
1

Scaffold the project

Creates n8c.config.json, a .env template, a .gitignore, and reconciles the database indexes.

n8c init
2

Fill in your environment

Edit .env with your connection details. These files are gitignored — secrets never get committed.

N8N_BASE=https://your-n8n.example.com
N8N_API_KEY=your-api-key
MONGO_URI=mongodb://localhost:27017
MONGO_DB=n8c
N8C_CREDENTIAL_ENCRYPTION_KEY=a-strong-random-key
3

Pull your instance into code

Imports every workflow, prompt and credential from n8n into the database and writes them to disk as readable files under n8c/.

n8c pull
4

Edit & commit

Change a workflow's apply.ts, tweak a prompt, review the diff in a pull request — normal git workflow.

git add n8c/ && git commit -m "Update triage prompt"
5

Preview and deploy

plan shows exactly what will change against live n8n; apply pushes it and records the new version. Use apply --force to plan and apply in one step.

n8c plan
n8c apply
💡

Promote the same files to production. Run n8c -e prod pull once on prod's fresh DB to build its credential mapping, then n8c -e prod plan and n8c -e prod apply. The env-neutral local IDs resolve to prod's real n8n IDs at push time.

Storage backends

MongoDB for teams, SQLite for zero setup

The database is n8c's source of truth for version history. Pick the one that fits — set it with database in n8c.config.json; the whole workflow is otherwise identical.

MongoDB

// n8c.config.json
{ "database": "mongodb" }

// .env
MONGO_URI=mongodb://localhost:27017
MONGO_DB=n8c

SQLite

// n8c.config.json
{ "database": "sqlite",
  "sqlite": { "file": "n8c.sqlite" } }

// .env — no database server needed
ℹ️

Why prompt-content needs MongoDB. The runtime prompt registry exists so an n8n node can read prompts from the database while a workflow runs — and n8n can only reach MongoDB for that. On SQLite that single feature is skipped; workflows, prompts, credentials, versioning and rollback are all identical.

How it works

Concepts

Everything n8c manages lives as a folder of code, versioned in the database.

Entities on disk

n8c/
├── workflows/<localId>/
│   ├── metadata.json   # name, description
│   └── apply.ts        # returns the workflow body
├── prompts/<localId>/
├── prompt-contents/<localId>/
└── credentials/<localId>/

apply.ts default-exports the entity body. Workflows export as readable code — each node a named object, connections inline. Local IDs are env-neutral UUIDs, never the n8n ID.

Database collections

  • n8c_workflows — live docs + every version
  • n8c_prompts — build-time prompts wired into nodes
  • n8c_prompt_contents — runtime prompts read at execution
  • n8c_credentials — encrypted at rest + versions
  • n8c_manifests — version metadata only
  • n8c_definitions — per-env localId → n8nId map

Every collection name is configurable — override the n8c_ prefix or any single name in n8c.config.json.

Reference

Commands

Every command self-documents via --help. Global flags: -e, --env <name> and --pipe.

n8c initScaffold config / .env / .gitignore and reconcile DB indexes.
n8c pulln8n + DB → files. --no-export = DB only.
n8c planDiff files vs live → write .states/n8c.state.<env>.json. --destroy includes deletes.
n8c applyExecute the saved plan (push to n8n, then commit the DB). --force = plan + apply in one step.
n8c listGeneration versions (releases), newest first — id, kinds covered, message. --full untruncates.
n8c createScaffold a workflow | prompt | credential | node.
n8c restore <generation>Roll every kind back to a generation version; --apply deploys it now.
n8c drop <generation…>Delete generation versions from history (each drops across all kinds; the active one can't be dropped).
n8c db exportDump all n8c collections to a backup file (MongoDB only).
n8c db importRecreate + re-import collections from a backup file (MongoDB only).
🔐

Secrets, done right. A credential's apply.ts reads its value from process.env.MY_TOKEN. The reference is what lives in files and the DB — never the resolved secret. At rest, credential data is encrypted with AES-256-GCM.
